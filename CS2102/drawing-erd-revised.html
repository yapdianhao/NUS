2. DOT Language

Most of the ER diagrams used in lectures/tutorials are drawn using the DOT language which is a specification language for drawing directed/undirected graphs.

There is a family of tools (e.g., dot, neato, circo, fdp, sfdp) that can process DOT specifications. These software are available on the stu server.

2.1. Example 1

The following is the DOT specification of the ER data model for Tutorial 3’s question 3(e).

// tutorial 3, question 3e

digraph q3e {
	newrank=true;
	overlap=false;
	splines=false;

	node [shape=ellipse]; pssn[label=<<u>ssn</u>>]; patname [label="name"]; age;  pataddress [label="address"];
	node [shape=box]; Patients;
	pssn -> Patients [dir=none];
	patname -> Patients [dir=none];
	age -> Patients [dir=none];
	pataddress -> Patients [dir=none];

	node [shape=diamond]; Has [label="Primary\nPhysician"];
	node [shape=box]; Doctors;

	Patients -> Has [color="black:white:black"];
	Has -> Doctors [dir=none, color="black:white:black"];

	node [shape=ellipse]; dssn[label=<<u>ssn</u>>]; dname [label="name"]; speciality; years;
	Doctors -> dssn [dir=none]; Doctors -> dname [dir=none]; Doctors -> speciality [dir=none]; Doctors -> years [dir=none];

	node [shape=ellipse]; date;
	node [shape=ellipse]; price;
	node [shape=ellipse]; qty;
	node [shape=diamond]; Prescribes;
	node [shape=diamond]; Sells;
	price -> Sells [dir=none];
	date -> Prescribes [dir=none];
	Prescribes -> qty [dir=none];
	node [shape=box]; Drugs [peripheries=2];
	Sells -> Drugs [dir=none];

	node [shape=box]; Supervisors;
	node [shape=diamond]; Supervises;
	node [shape=box]; Pharmacies ;

	node [shape=diamond]; Makes [peripheries=2];
	Doctors -> Prescribes [dir=none];
	Prescribes -> Drugs [dir=none];

	node [shape=ellipse]; sssn[label=<<u>ssn</u>>]; sstartDate [label=<<u>startDate</u>>];
	Supervises -> Supervisors [dir=none] ;
	Supervisors -> sssn [dir=none];
	Supervises -> sstartDate [dir=none] ;

	node [shape=box]; Companies [label="PharmaceuticalCompanies"];

	node [shape=diamond]; Contracts;

	Companies -> Contracts [dir=none];

	node [shape=ellipse]; paddress [label="address"];
	node [shape=ellipse]; cstartDate [label="startDate"]; endDate; text;

	Pharmacies -> Contracts [dir=none, color="black:white:black"];

	node [shape=ellipse]; pcname [label=<<u>name</u>>]; pcphone [label="phone"];
	node [shape=ellipse]; pname [label=<<u>name</u>>]; pphone [label="phone"];
	Companies -> pcname [dir=none];
	Companies -> pcphone [dir=none];
	paddress -> Pharmacies [dir=none];
	endDate -> Contracts [dir=none];
	Contracts -> text [dir=none];
	Contracts -> cstartDate [dir=none];

	Drugs -> Makes [color="black:white:black"];
	Makes -> Companies [dir=none];
	node [shape=ellipse]; tradeName[label=<<u>tradeName</u>>]; formula;
	Drugs -> tradeName [dir=none];
	Drugs -> formula [dir=none];
	Pharmacies -> Sells [dir=none, color="black:white:black"];

	Pharmacies -> pname [dir=none];
	Pharmacies -> pphone [dir=none];

	Supervises -> Contracts [arrowhead=box, color="black:white:black"];

	// fontsize=22;
}

The above specification (in a file named q3e.dot) can be processed into an image file named q3e.png using the dot command as follows:

$ dot -Tpng:cairo -o q3e.png q3e.dot

3. ERD

ERD is a specification language that is specifically designed for drawing ER diagrams that is based on the DOT language.

You can process an ERD specification using the tool named erd which is available on the stu server.

The full pathname of the erd command is /home/course/cs2102/bin/erd.

To avoid typing this long pathname, append the following line to your ~/.bashrc file:

export PATH=/home/course/cs2102/bin:$PATH

Append the following line to the file ~/.profile if it is not already there:

source ~/.bashrc

You should be able to just type erd to invoke the program when you relogin to the stu server.

3.1. Grammar

The following is the grammar of the ERD language. Terminals are shown in uppercase and nonterminals in lowercase. Literal characters are given in single quotes. Parentheses ( and ) indicate grouping when needed. Square brackets [ and ] enclose optional items. Vertical bars | separate alternatives. The keywords (e.g., ENT, REL) are case-insensitive.

erd : objs
objs : obj [ objs ]
obj : entity | weakentity | relationship | isa
entity : ENT name attrprops
weakentity : WENT name wentprops IREL name [ATT attrs]
relationship : REL name relprops
isa :  ISA isaprop superclass subclasses
wentprops : wentprop [ wentprops ]
wentprop : attrprop | OENT name [ constraint ]
attrprops : attrprop [ attrprops ]
attrprop : (KEY|ATT) attrs
relprops : relprop [ relprops ]
relprop : PAR relpars | attrprops
relpars : relpar [ relpars ]
relpar : name ([ constraint [ '"'role'"' ] ] | [ '"'role'"' [ constraint ] ])
constraint : <=1 | >=1 | =1
isaprop : o=val c=val | c=val o=val
superclass : name
subclasses : subclass [ subclasses ]
attrs : attr [ attrs ]
attr : name
subclass : name
name:  identifier
role:  identifier
val : 0 | 1

Each ERD specification consists of a collection of objects. Each object is an entity (ENT), a relationship (REL), a weak entity (WENT), or a ISA hierarchy (ISA).

An entity (ENT) has a name and at least one key attribute (KEY) and zero or more non-key attributes (ATT).

A relationship (REL) has a name and at least one participant (PAR). Each relationship participant is an entity, a relationship, or a weak entity. Each relationship participant has an optional rolename (enclosed in double quotes) and an optional relationship constraint. A relationship constraint is either a key constraint (<= 1), total participation constraint (>= 1), or both key and total participation constraints (= 1). Each relationship can have zero or more attributes (KEY/ATT).

A weak entity (WENT) has a name, the name of the owner entity (OENT), at least one partial key attribute (KEY), zero or more non-key attributes (ATT), and the name of the identifying relationship (IREL). An owner entity name can optionally be followed by a relationship constraint.

A ISA hierarchy (ISA) consists of its covering constraint, its overlap constraint, the name of its superclass followed by one or more subclass names. A covering constraint is either c=1 (if it is satisfied) or c=0 (if it is not satisfied). An overlap constraint is either o=1 (if it is satisfied) or o=0 (if it is not satisfied).

An identifier is a sequence of characters that matches the regular expression [A-Za-z_][A-Za-z_0-9]*.

A comment must start at the beginning of the line with two forward slash characters (//).

3.2. Example 1

The following is the ERD specification of the ER data model for Tutorial 3’s question 3(e).

// Tutorial 3, question 3e

ENT Patients
KEY ssn
ATT name age address

ENT Doctors
KEY ssn
ATT name years speciality

ENT Pharmacies
ATT phone address
KEY name

ENT PharmaceuticalCompanies
KEY name
ATT phone

WENT Drugs
KEY tradename
ATT formula
OENT PharmaceuticalCompanies
IREL Makes

REL PrimaryPhysician
PAR Doctors Patients =1

REL Prescribes
ATT date qty
PAR Doctors Drugs

REL Sells
PAR Pharmacies >=1 Drugs
ATT price

REL Contracts
PAR PharmaceuticalCompanies Pharmacies >=1
ATT text startdate enddate

ENT Supervisors
KEY ssn

REL Supervises
PAR Supervisors
PAR Contracts
KEY startTime

The above specification (in a file named q3e.erd) can be processed using the erd command as follows:

$ erd q3e.erd

The above command generates 6 output files:

  • tmp-q3e.dot (the translated DOT specification file)

  • q3e-circo.png (the image produced by circo)

  • q3e-dot.png (the image produced by dot)

  • q3e-fdp.png (the image produced by fdp)

  • q3e-neato.png (the image produced by neato)

  • q3e-sfdp.png (the image produced by sfdp)

The different programs apply different heuristics to automate the layout of the ER diagram.
